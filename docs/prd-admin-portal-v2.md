# Product Requirements Document: 管理者ポータル機能拡充

**Author**: shigetaokito
**Date**: 2026-03-30
**Status**: Draft
**Stakeholders**: 塾講師（admin/teacher）、塾長（superadmin）、開発チーム

---

## 1. Executive Summary

総合型選抜コーチングアプリ「Coach」の管理者ポータルは、生徒が生成する8種類のデータのうち3種類しか可視化できていない。講師が生徒の出願準備の全体像を把握し、適切なタイミングで介入するために、**出願書類の進捗管理**、**面接練習の可視化**、**活動実績の確認**、**生徒総合ビュー**の4機能を追加する。

---

## 2. Background & Context

### 問題の本質
総合型選抜の指導では、小論文だけでなく**志望理由書・活動報告書・面接対策**が合否を分ける。現状の管理者ポータルは小論文添削とセッション管理に偏っており、講師は以下ができない：

- 生徒が志望理由書をどこまで書いたか分からない（出願期限に間に合うか不明）
- 面接練習をどれだけやったか、スコアが伸びているか見えない
- 活動実績の入力状況が分からない（面談で初めて「まだやっていない」と判明する）
- 生徒の全体的な準備状況を一覧で把握できない

### 現状の管理者可視性

| データ種別 | 生徒側 | 管理者可視性 | 指導への影響 |
|-----------|--------|-------------|------------|
| 小論文添削 | ✓ | ✓ 完全 | — |
| 模擬面接 | ✓ | ✗ なし | **面接指導の方針が立てられない** |
| 活動実績 | ✓ | ✗ なし | **出願書類への反映が遅れる** |
| 出願書類 | ✓ | ✗ なし | **期限管理ができない（致命的）** |
| セッション | ✓ | ✓ 完全 | — |
| 自己分析 | ✓ | ✗ なし | 面談材料として有用 |
| 性格診断 | ✓ | ✗ なし | 参考情報レベル |
| 弱点/成長 | ✓ | △ 部分的 | コホート分析不可 |

## 3. Objectives & Success Metrics

### Goals
1. 講師が**全生徒の出願準備進捗**を一画面で把握できる
2. 講師が**面接・書類・活動の各データ**に生徒詳細画面からアクセスできる
3. 出願期限に対する**書類完成度アラート**が自動で上がる

### Non-Goals
1. 生徒向けUIの変更（今回は管理者ポータルのみ）
2. 新しいデータ収集（既存の生徒データを管理者に見せるだけ）
3. BigQuery分析の拡充（既存のsuperadmin分析で十分）
4. 自己分析・MBTI結果の管理者ビュー（指導に直結しない。面談時に口頭確認で十分）

### Success Metrics

| Metric | Current | Target | Measurement |
|--------|---------|--------|-------------|
| 管理者が閲覧可能なデータ種別 | 3/8 (38%) | 7/8 (88%) | 実装カウント |
| 期限切れ書類の発生率 | 不明（管理不能） | 管理者が全件把握可能 | アラート件数 |
| 面談前の情報収集時間 | 面談内で確認 | 事前に画面で確認済み | 講師ヒアリング |

## 6. Solution Overview

### アーキテクチャ方針
- **既存データの読み取りのみ**。新しいFirestoreコレクションは作らない（アラート用の1フィールド追加のみ）
- 既存の `requireRole()` + `managedBy` スコーピングを全新機能に一貫適用
- 生徒詳細ページにタブを追加する形で拡張（既存ページの改修）
- API は既存の `/api/admin/students/[id]` を拡張、または新規エンドポイント追加

### アクセス制御モデル

| ロール | 見える生徒 | 見える講師 | 用途 |
|--------|-----------|-----------|------|
| **teacher** | 自分の`managedBy`生徒のみ | — | 日常の生徒指導 |
| **admin** | 教室の全生徒 | 教室の全講師 | 教室運営・講師管理 |
| **superadmin** | 全生徒 | 全講師・全管理者 | 全体統括 |

既存の `requireRole()` + Firestoreの `managedBy` フィルタで実現済み。新規APIも同じパターンを適用。

### 主要な実装アプローチ

**P0-1: スコーピング徹底**
- 全新規API/ページで `requireRole()` + `managedBy` フィルタを適用
- admin は配下の全teacher の生徒も表示可能

**P0-2: 生徒一覧の志望校表示**
- `/api/admin/students` のレスポンスに `targetUniversities` の大学名を含める
- テーブルに志望校カラム追加（第一志望 + 「他N校」バッジ）
- 志望校フィルタ（ドロップダウン）追加

**P0-3〜P0-5: 生徒詳細タブ追加**
- `/admin/students/[id]` に3つのタブを追加（書類/面接/活動）
- 各タブはFirestoreの `users/{id}/documents`, `users/{id}/interviews`, `users/{id}/activities` を読み取り
- スコア推移は既存の Recharts パターンを再利用

**P0-6: 書類期限アラート**
- 既存の `/api/admin/alerts` に `document_deadline` タイプを追加
- アラート判定ロジック: 書類の `deadline` と現在日時を比較、完成度が100%未満の場合にアラート生成

**P1-1: 進捗一覧ページ**
- 新規ページ `/admin/progress`
- 全担当生徒の横断ビュー: 各項目の最新状態をバッジで色分け表示
- 志望校カラム含む一覧表示

### プロプラン機能のアーキテクチャ

**Stripe連携（PP-2）**
- `stripe` npm パッケージ + Stripe Webhook
- Checkout Session → 支払い完了 → Webhook → Firestore `users/{id}.plan = "pro"` に自動更新
- Customer Portal でプラン変更・解約を生徒/保護者が自己管理
- 管理者画面からも手動でプラン切替可能（トライアル等）

**スケジュールグリッド（PP-3〜PP-6）**
- Firestoreスキーマ:
  - `users/{teacherId}/availability`: `{ dayOfWeek: number, startTime: string, endTime: string }[]`
  - `users/{studentId}/preferredSlots`: `{ dayOfWeek: number, startTime: string, endTime: string }[]`
  - `assignments/`: `{ teacherId, studentId, dayOfWeek, startTime, endTime, createdAt }`
- UI: Apple Calendar風の週間グリッド（30分単位、7:00-21:00）
  - 緑セル = 講師空きコマ
  - 青セル = 生徒希望
  - ハイライト = 両方が重なるマッチ枠
  - クリック → アサイン確定モーダル
- アクセス制御: Proプラン生徒のみグリッドに表示

## 8. Timeline & Phasing

### Phase A: ユーザー管理基盤（P0-1〜P0-3）— 2-3日
運用に必須。これがないとsuperadminに毎回依頼が必要。
1. 管理者向け生徒追加ページ（`/admin/students/new`）+ API
2. 管理者向け講師管理ページ（`/admin/teachers`）+ 招待コード発行
3. スコーピング適用確認（全新規機能）

### Phase B: 生徒一覧強化 + プロフィール編集（P0-4, P0-9）— 1-2日
1. 生徒一覧に志望校カラム・フィルタ追加
2. 生徒詳細にプロフィール編集機能追加

### Phase C: 生徒詳細タブ拡充 + コンテンツ閲覧（P0-5〜P0-7, P0-10〜P0-11）— 4-5日
1. 出願書類タブ（API + UI）
2. 面接履歴タブ + 会話ログ閲覧（API + UI）
3. 活動実績タブ（API + UI）
4. 小論文元テキスト閲覧（既存添削履歴を拡張）
5. 面接個別詳細ページ（会話ログ + スコア + フィードバック）

### Phase D: アラート + 合否トラッキング（P0-8, P0-12）— 2-3日
1. 書類期限アラートのロジック追加 + UI対応
2. 受験結果記録UI（生徒詳細に追加）
3. 合格率ダッシュボード集計

### Phase E: 進捗一覧 + 詳細ビュー + 設定（P1-1〜P1-6）— 3-5日
1. 進捗マトリクスページ（志望校含む）
2. 面接詳細ビュー・書類詳細ビュー
3. 自己分析タブ
4. 生徒非活動化・管理者設定ページ

### Phase F: Stripe課金連携（PP-1〜PP-2）— 2-3日
1. Stripe Checkout + Webhook連携
2. 生徒プロフィールにplanフィールド追加
3. 管理者画面でプラン手動切替UI
4. Customer Portal連携

### Phase G: スケジュールグリッド（PP-3〜PP-6）— 4-5日
1. 講師の空きコマ登録UI（講師設定画面）
2. 生徒の希望時間帯入力UI（生徒設定画面、Pro限定）
3. Apple Calendar風マッチンググリッド（管理者画面）
4. アサイン確定・変更・キャンセル機能
5. アサイン済みスケジュール一覧ビュー

### 依存関係
```
Phase A（ユーザー管理）→ Phase B（一覧+編集）→ Phase C（タブ追加）→ Phase D（アラート）
                                                              → Phase E（一覧+詳細+設定）

Phase F（Stripe課金）→ Phase G（スケジュールグリッド）
                      ※Phase F/GはPhase A-Eと並行可能
```
Phase Aが最優先。Phase F→Gはプロプラン機能として独立して進行可能。
